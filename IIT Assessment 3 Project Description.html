Detailed Description

Aims

In this project, the goal that our team members want to achieve is improve the user aiming skill. However, there were already some similar programs on the market. Our team desired to create a perfect project which has more detailed and features more analysis than the current program on the market. To be more specific, we want to conduct a survey to collect the data about:

  • speed (average reaction and a log of per shot reactions)
  • precision (based off distance from centre of target
  • accuracy (hit shots divided by total shots)
  • directional weaknesses

After collecting all the data, our team will compare to the users average score. From that result, we will know what the users really need to improve their aim. And based on that we will add in our system more functions to make it perfect.

Plans and Progression

Our project has the aim to help others improve their aim. We will achieve this by giving the user an application that allows them to practise their aim in short high intensity tasks, similar to a HIIT (high intensity interval training) workout. This is done through a short 60 second task where a user has the intention of achieving a high score, calculated by the amount of targets shot minus the amount of times the user missed, although the scoring system will become improved over time as more functions are implemented to differentiate between similar scores. Currently while developing a prototype application the current functions include score, shots missed and a countdown timer.

Upon opening this application the user is greeted with a main menu featuring a “play” button, “options” button and “quit” button which all currently function. Although upon pressing play the game level lighting appears different compared to game scene whilst editing it, this creating an obstacle which will have to be overcome. Once this project is completed, there should be a range of tasks to target specific skills required within the game for the user to improve, an accurate scoring system and smooth transitioning between game, menu and end screen.

As many of our group members enjoy playing video games, and some have also used similar existing programs we decided that this would be a good project to create. Although there's already free ones, we aim to provide more features, stats and customizability than those, whilst aiming to create one better than paid applications. To achieve this task must be created that focus on specific skills to ensure that the user is able to improve through muscle memory and intensive work within those areas, as well as being able to provide statistics of each attempt to show improvement and what still needs to be improved.

The plan to create this is to separate the application into smaller separate tasks (main menu, game scene, score screen) and within each scene, create smaller tasks to eventually achieve a fully functional app. To begin I would create a main menu, over time this can be improved, but for the sake of the prototype it would have a play button, options button and quit button. The play button would change the scene so the user is able to play the game, options would give the user their own customisability and finally quit would close the application. After this was done the game scene would be created, starting by adding the character and camera movement, sensitivity adjustment and a weapon for the user to hold. Once the character is complete then there would have to be targets for them to shoot, done by using spheres and making them respawn in a random location once shot, which would also require the user the ability to shoot. Once shooting and respawning is done then there would need to be a countdown and thcore and misses counter, as well as a return to menu so the user is able to leave mid game if they’d like. To complete the functionality of the app there would be an end screen which would allow the user to play again or close the application.

Progress:

Main menu - To begin the main menu an image was imported and changed to a sprite (2D and UI) and a canvas was added to the unity scene, as well as a panel so that the image can be added as the background.

Then a few buttons were added so that the user can change scenes, open options and close the game. For the play button a PlayGame() function was added to change the scene on button press, then the quit button was set to a QuitGame() function that would close the application and lastly the options button which would require the most work out of the three main menu buttons. This required another menu (in the same scene) which would be disabled and enabled based when the buttons are pressed to show the options menu, as well as close it.

Game scene - Now that the prototype menu is done a new scene for the game was added and also added into the build settings. This was the game scene, upon creating it the first things added were a few walls and a floor that could be used as a basic area. After this a cylinder was added to the scene and the camera was placed in it so that it would act as the user. For the user to be able to look around a PlayerScript was added to the player object, in this it would lock the users cursor to the middle of the screen as well as make it invisible so that the user could aim and stay tabbed into the game. To allow the player to look around a script was written to get the mouse inputs of the X and Y axis using Vector3, as well as clamping the Y axis 90 and -90 degrees so the user can’t look 360 degrees vertically.

In order to shoot targets there would need to be both a crosshair and a method of shooting. This was done by adding a canvas to the scene and a small square image centered in the canvas to act as the crosshair (which would later become customisable). To solve the shooting a new script was added to create a raycast when mouse1 was pressed.

The target was then added as a sphere so that the raycast could be tested. When the target was hit, a script was then made to respawn the target in a random location, defined within an area made using an empty object with a box collider (disabled) and a script to find a random value for each the X, Y and Z axis within the box collider and reposition the target there.

Now that both the players ability to shoot and the target was made, within the raycast script, another few lines of code were added to check if the raycast hit a target, if a target was hit then a textbox would update to shot the amount of hits, else if the raycast did not collide with a target then a text box for misses would be updated and this would show the user the amount of hits and misses they had. To create a scoring system each target would give the user 10 points, each miss would minus 5 points and the end amount of misses would also be minused from the total, to prevent negative scores, if the score was less than or equal to 0 then the score would equal 0.

End scene - finally to finish the prototype the last scene was added to display scores and allow the user to play again or quit. This was done by once again adding a canvas and creating a background to the scene, as well as a button to change scene back to the previous (game) one to play again, with an option to quit the game. For the main purpose of this scene (to display scores) a few text boxes were added to show the hits, misses and final score. These were values updated whilst in game and were called upon during this scene to display for the user.

This is the current progress for the prototype, all functions currently working and the game is playable. Roadblocks along the way were many coding issues that were resolved over time by re-reading the code or trying to change it. There were issues with lighting whilst testing the game within the unity engine but once the game was built these issues went away.

Hopefully with more testing more issues will be discovered so that they may be fixed to iron out the quality of the game and as we progress further more tasks will be added to give the user more options than the current prototype.

Scope and Limits

This project is all about an ‘Aim Training’ software. This will train new or entry level gamers to improve their target in First-Person Shooter games such as Counter Strike: Global Offensive, Fortnite and many more. For the available time and introductory period, we would only include a limited amount of options and setting in the system. At this stage, we would only make this available on the web.Also, the type of firing will be only one shot per click. Later on we would add more and more options(Which would be described in further part of this section) to it.

At this stage the features included in it will be:

  • Options to adjust sensitivity of mouse.
    This option would enable the user to increase or decrease the speed of target cross in proportion to the movement of the mouse.
  • Option to adjust speed, colors and size of targets
    Changing this setting will enable the user to adjust the movement speed of targets on which the user will shoot. Users can increase or decrease the flashing and emerging speed of targets . In addition to that, colors and size of targets can be adjusted according to the user's comfort.
  • Settings for tweaking field of view.
    Users will be able to zoom in and out the screen for hitting aims.
  • Change the color of the crosshair.


Users will be able to see the following data after using this Aim Trainer:

  • Speed or reaction time.
    As soon as the target appears on the screen, the time will be calculated between the flashing of the target and shooting it. When you quit the game, the average time of shooting each target will be recorded.

  • Precision
    As soon as you hit a target, the data will be recorded for the distance between the place where you shot and the centre of the aim. At the end, you will get precision data including:
    1) How many shots you had hit on the centre.
    2) How many of them were just mere hits, that means how many targets you hit were just shot on the edge.
    3) Average distance of targets hit from centre.

  • Accuracy
    The count of unnecessary gunshots or the missed gunshots. This will be represented in percentage form, calculated by shots (hit/Total number of shoots attempted)*100.

  • Hand stability
    The amount of unneeded movement of hand or mouse between two shots. For example, after attempting a shot your pointer moves here and there which could be avoided to increase precision and efficiency.

  • Tips and guidances
    After playing a considerable number of times, after each gameplay user will be provided a tip to improvise his scores.


On the homescreen, the user can access the following things:

  • Leaderboard
    In this section, the user can have a look at some of the best scores around the world. Over here he can see some clips of some of the phenomenal game plays of leaderboard toppers.
  • Friends
    The user can have a look at scores of his friends to increase a sense of competition.

For the deliverables or if someone asks to demonstrate our project, it will be limited to above mentioned features. We will be demonstrating a highly refined as well as user friendly version of the system.
Refinement and improvement will be done on the bases of user testing and feedback which will be discussed in the upcoming sections.


If we talk about the parts or features which wouldn’t be available at this stage will be:

  • Game specific backgrounds and ammunation firing type
    There will be only a common firing background. No different shooting scenarios would be available for a variety of games.

Testing

User Testing is one of the most crucial processes during development of these kinds of projects. We would perform tests at small intervals of time.

Our first test will be at a stage when we complete building a prototype. At this stage, we would be able to feed in necessary features of our project. But at this particular point in time, our main focus would be to test the main goal of our project, i.e. aim improvising assistant, rather than focusing on the design and usability of the game.
First of all, we would identify the potential users and analyse how we could bring out the best by testing different user groups. Initially, we would test our project through 2 different groups of people, pro-gamers and entry-level gamers.

The first group would be some of the champions of e-sports, especially ones who have mastered first-person shooter games. We would hand over our prototype to them and record the average time they play it. We would try to improvise our project by asking them open-ended questions. This would be helpful because they would be the perfect judges of this project as they have mastered this section of aiming.
To understand, let us use a hypothetical situation. Suppose we approach a pro gamer to test our project. During his play, he could perfectly understand if the movement of targets really imitated the movement of enemies in the FPS games. According to their reviews, we could make tweaks into it.
Some of the questions that will be included in our testing would be:

  • Are there any other types of statistics which you would like to see in this prototype?
  • According to you, what could be some changes during the gameplay you would like to have which would be more fruitful?

During testing of this user group, our motto will be to increase average gameplay time of the people we test. This will indicate to us whether or not our project really makes gamers stick to their gaming chairs for a long time. There would be a constant effort by us to increase this average gameplay time by constantly making changes based on feedback of group 1.

Once we come to know that after making several changes, the average play time has become constant, we would move onto testing the next group of testers, which are ‘The entry level gamers’. This test would be a longer test as compared to testing of the first user group.

While testing the second groups, our main motive would be to observe to what extent our project improvises the aim of the users. Because at the end of the day, this is what matters. For example, if we gradually increase the randomness and speed of the targets rather than just simply increasing the speed of the targets and the results were more fruitful than the previous tests, we would implement it.
For this we would make use of ‘Usability Tests’.

After this stage, we would have achieved a good amount of perfected completions of user expected tasks. we would then proceed towards the usability and design of the system.
For this we would make use of ‘Usability Tests’. In this process we would ask potential users to complete certain tasks and record the timings for it. We would compare the timings of the different models in which the buttons and adjustments would be at different places on the screen. For an instance, we have 3 different models in which the only difference would be button placements. The model which offers the least average timings of completion of tasks will be finalised. At the same time, during designing the interface, Nielsen’s Heuristics and Usability principles will be kept in mind.

Risks

There are many health risks with this project. When training, the user may become motion sick from constantly moving the camera. This will most likely be a settings problem with the game itself as motion sickness is caused by a wrong field of view (FOV), but the application will most likely have to place a warning before its use.

Carpal Tunnel Syndrome is another risk as the user will be repeating very similar movements in order to train their aim. As a result, the repetition and increasing intensity and stress of perfecting their aim may cause damage to the muscles in their wrists. This can be avoided by proper aiming with the mouse. Rather than flicking with the wrists, the user should be instructed to move their arm to move the mouse. This can also be a problem as it may lead to “tennis elbow” or inflammation of the tissue connecting the forearm muscle and the elbow. Treatment is simply rest so placing a warning after a while of use may be necessary. While “tennis elbow” is less detrimental to the user than Carpal Tunnel Syndrome, it is still a risk to take into consideration.
If the user is playing on console, there is the risk of “Gamer’s thumb”. Also known as “De Quervain Tenosynovitis”, it is the inflammation of the thumb side of the wrist as a result of repetitive and excessive use of the thumbs. Like “tennis elbow”, it is treated by resting.
The user may also have eye strain as a result of watching or staring at the screen for long periods of time. This can lead to headaches, poor concentration, and damage to the eyes resulting in the user needing to wear glasses. A notification will need to be implemented in order to allow the user to rest their eyes after long periods of use.

There are also some psychological risks associated with gaming in general and as such aim training. Gaming addiction or internet gaming disorder (IGD) may lead the user to have lower sociability skills, withdrawal, increased dopamine threshold which can lead to decreased pleasure when completing other tasks or eating food leading to obesity and the need to play more in order to meet the dopamine threshold. If an online leaderboard is implemented into the application, this could lead to the user developing a lower mental self-image as a result of comparing scores with other players.
